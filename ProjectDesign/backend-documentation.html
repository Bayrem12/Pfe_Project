Principe de la règle de dépendance

Domain (noyau)

Aucune dépendance externe. Contient uniquement des entités, interfaces et énumérations métier. C'est le cœur stable du système.

Application

Dépend uniquement de Domain. Contient la logique applicative via CQRS. Ne connaît ni EF Core, ni ASP.NET.

Persistance

Implémente les interfaces définies dans Application/Domain. Gère EF Core, les migrations, les repositories concrets.

API

Point d'entrée HTTP. Dépend de Application via MediatR. Ne contient aucune logique métier — délègue tout aux handlers.

DashboardRepository — Requêtes d'agrégation

Exemple de la complexité du repository dashboard (lecture multi-tables, server-side via EF Core) :

// Calcul du pass rate global — tout en SQL serveur via EF Core
var totalResults = await _context.TestResults
    .AsNoTracking()
    .CountAsync(r => !r.IsDeleted, cancellationToken);

    .AsNoTracking()
    .CountAsync(r => !r.IsDeleted && r.Status == TestStatus.Passed, cancellationToken);

Toutes les requêtes du DashboardRepository utilisent .AsNoTracking() (lecture seule) et des projections .Select() directes vers les DTOs, évitant de charger des entités complètes en mémoire. Les sous-requêtes complexes sont exécutées entièrement côté serveur SQL.

Vérification des permissions (ProjectsController)

Étant donné que le rôle n'est pas dans le JWT, chaque endpoint protégé effectue une requête DB supplémentaire pour vérifier le rôle :

private async Task<string> GetCurrentGlobalRoleAsync()
{
    var roleName = await _dbContext.Users
        .AsNoTracking()
        .Where(u => u.Id == CurrentUserId)
        .SelectMany(u => u.UserRoles)
        .Select(ur => ur.Role.Name)
        .FirstOrDefaultAsync();
    return (roleName ?? string.Empty).Trim().ToLowerInvariant();
}

private async Task<bool> CanReadProjectAsync(Guid projectId)
{
    var role = await GetCurrentGlobalRoleAsync();
    if (role == "admin") return true;

    // Manager/Tester/Viewer : uniquement si membre du projet
    return await _dbContext.ProjectMembers
        .AsNoTracking()
        .AnyAsync(m => !m.IsDeleted && m.ProjectId == projectId && m.UserId == CurrentUserId);
}

Pattern CQRS et pipeline MediatR

Séparation Commands/Queries, pipeline de validation FluentValidation, profils AutoMapper et convention inner-class handler.

Principe CQRS (Command Query Responsibility Segregation)

Chaque opération est soit une Command (modifie l'état) soit une Query (lecture seule). MediatR achemine l'objet vers son handler unique.

Commands

Mutent l'état du système. Ex: AddProjectCommand, DeleteScenarioCommand, AddUserLogin

Queries

Lecture seule. Ex: GetAllProjectsQuery, GetProjectByIdQuery, GetDashboardSummaryQuery

Convention de nommage

Handler défini comme inner class dans la Command/Query : AddProjectCommand.AddProjectCommandHandler

Découplage

Les controllers ne contiennent aucune logique métier — ils appellent uniquement _mediator.Send(cmd)

Structure d'une Command

// Pattern typique : Command + Handler comme inner class
public record AddProjectCommand(
    string Name,
    string Description,
    Guid   CreatedById
) : IRequest<ResponseHttp>
{
    public class AddProjectCommandHandler : IRequestHandler<AddProjectCommand, ResponseHttp>
    {
        private readonly IProjectRepository _repository;

        public AddProjectCommandHandler(IProjectRepository repository)
            => _repository = repository;

        public async Task<ResponseHttp> Handle(
            AddProjectCommand request,
            CancellationToken cancellationToken)
        {
            // Logique métier ici
            var project = new Project { Name = request.Name, ... };
            await _repository.AddAsync(project);
            return new ResponseHttp { Status = 200, Resultat = project.Id };
        }
    }
}

Pipeline MediatR — ValidationBehavior

Avant d'atteindre le handler, chaque requête MediatR passe par le ValidationBehavior qui :

  1. Récupère tous les validators FluentValidation associés au type de requête
  2. Exécute tous les validators
  3. Si des erreurs existent, lève une ValidationException avec le message concaténé
  4. Si tout est valide, appelle le handler suivant dans le pipeline
// ValidationBehavior.cs
public class ValidationBehavior<TRequest, TResponse> : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IRequest<TResponse>
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;

    public async Task<TResponse> Handle(TRequest request, RequestHandlerDelegate<TResponse> next, ...)
    {
        var failures = _validators
            .Select(v => v.Validate(request))
            .SelectMany(r => r.Errors)
            .Where(f => f != null)
            .ToList();

        if (failures.Count > 0)
            throw new ValidationException(failures);

        return await next();
    }
}

IAAgentService — Service HTTP typé

// Persistance/Services/IAAgentService.cs
public class IAAgentService
{
    private readonly HttpClient _httpClient;
    private readonly TestAutoumatisationContext _context;

    // Timeout de 5 minutes pour l'exécution des tests IA
    public IAAgentService(HttpClient httpClient, IConfiguration config, ...)
    {
        httpClient.BaseAddress = new Uri(config["IAAgent:BaseUrl"] ?? "http://localhost:8000");
        httpClient.Timeout     = TimeSpan.FromMinutes(5);
        _httpClient = httpClient;
    }

    public async Task<TestExecution> RunScenarioAsync(Guid scenarioId)
    {
        // 1. Appel HTTP à l'agent Python
        var response = await _httpClient.PostAsJsonAsync("/run", new { scenarioId });
        response.EnsureSuccessStatusCode();
        var agentResult = await response.Content.ReadFromJsonAsync<AgentRunResult>();

        // 2. Construire les entités de persistance
        var execution = new TestExecution
        {
            ScenarioId = scenarioId,
            Status     = ExecutionStatus.Completed,
            StartedAt  = agentResult.StartTime,
            Duration   = agentResult.Duration
        };

        // 3. Persister en base de données
        _context.TestExecutions.Add(execution);
        await _context.SaveChangesAsync();
        return execution;
    }
}

Audit Log Middleware

Le middleware AuditLogMiddleware intercepte toutes les requêtes HTTP de modification (non-GET) après leur traitement :

// AuditLogMiddleware.cs — Post-response middleware
public async Task InvokeAsync(HttpContext context)
{
    await _next(context); // ← Traitement de la requête d'abord

    // Conditions de journalisation :
    // - Requête authentifiée (userId présent)
    // - Pas une requête GET (lecture seule)
    // - Pas une erreur serveur (code < 500)
    // - Pas l'endpoint audit-logs lui-même (anti-récursion)
    if (!ShouldLog(context)) return;

    // Extraction de l'entité depuis le chemin URL
    var entityType = ExtractEntityType(context.Request.Path);

    using var scope = _serviceScopeFactory.CreateScope();
    var dbContext = scope.ServiceProvider.GetRequiredService<TestAutoumatisationContext>();

    dbContext.AuditLogs.Add(new AuditLog
    {
        UserId     = userId,
        Action     = context.Request.Method,
        EntityType = entityType,
        Timestamp  = DateTime.UtcNow,
        IpAddress  = context.Connection.RemoteIpAddress?.ToString()
    });
    await dbContext.SaveChangesAsync();
}

public class ResponseHttp
{

⚠️
Limitation architecturale — ResponseHttp non générique

ResponseHttp utilise object Resultat (non générique), ce qui sacrifie la sécurité de typage à compile-time. La convention de nommage Fail_Messages avec underscore ne respecte pas les conventions C# (PascalCase attendu : FailMessages). La solution serait d'utiliser ResponseHttp<T> générique.

Analyse critique et limitations connues

Revue honnête des faiblesses architecturales, de sécurité et de code — avec pour chaque point une solution recommandée.

Voici une analyse honnête des limitations et points d'amélioration identifiés dans le backend. Leur connaissance est indispensable pour la soutenance PFE.

Critique Rôle absent du JWT — requête DB par endpoint protégé
Le JWT ne contient que l'email et l'userId. À chaque appel d'un endpoint protégé, une requête SQL est lancée pour récupérer le rôle via la chaîne Users → UserRoles → Roles → Name. Pour une plateforme à fort trafic, ce pattern génère un goulot d'étranglement significatif et réduit les performances de toutes les routes protégées.
Solution recommandée : Inclure le claim role dans le JWT à la génération. Pour gérer les changements de rôle en temps réel, ajouter un timestamp role_assigned_at et le comparer à la date de la dernière modification côté serveur, ou utiliser un cache Redis (TTL 5 min).
Critique Refresh Token non implémenté — HTTP 501 Not Implemented
Le endpoint POST /api/auth/refresh retourne systématiquement HTTP 501. Les tokens JWT expirent sans possibilité de renouvellement automatique. Les utilisateurs sont déconnectés à l'expiration et doivent manuellement se reconnecter, nuisant à l'expérience utilisateur.
Solution recommandée : Implémenter un refresh token opaque (UUID v4) stocké haché en base de données. À chaque usage, renouveler le refresh token (rotation) pour détecter les vols de tokens. Mettre en liste noire les anciens tokens.
Majeur ResponseHttp non générique — perte de type-safety
public object Resultat { get; set; } — le champ Resultat est de type object. La sérialisation JSON produit des réponses inconsistantes selon les endpoints. La propriété Fail_Messages viole les conventions de nommage C# (PascalCase attendu).
Solution recommandée : Passer à ResponseHttp<T> générique. Renommer Fail_Messages en FailMessages. Utiliser JsonNamingPolicy.CamelCase globalement.
Majeur EnableSensitiveDataLogging activé inconditionnellement
optionsBuilder.EnableSensitiveDataLogging(true) est activé dans DbContextBase sans condition d'environnement. Cette option logue les valeurs des paramètres SQL (mots de passe, données personnelles), créant un risque RGPD et de sécurité si les logs sont accessibles en production.
Solution recommandée : Conditionner à l'environment : if (env.IsDevelopment()) optionsBuilder.EnableSensitiveDataLogging();
Majeur Typo dans le nom du fichier AuthController
Le fichier s'appelle AuthController..cs (double point). Visual Studio l'accepte sous Linux/ext4 mais crée des ambiguïtés et des problèmes potentiels sur les systèmes de fichiers Windows (NTFS) où le double point est un caractère réservé.
Solution recommandée : Renommer en AuthController.cs avec git mv pour conserver l'historique.
Majeur Pas de politique JSON camelCase
Aucune configuration JsonNamingPolicy.CamelCase dans Program.cs. Les réponses JSON utilisent PascalCase (Resultat, Status) alors que les conventions JavaScript et les frameworks frontend (Angular) attendent du camelCase. Le frontend doit s'adapter aux deux casses.
Solution recommandée : Ajouter .AddJsonOptions(o => o.JsonSerializerOptions.PropertyNamingPolicy = JsonNamingPolicy.CamelCase) dans Program.cs.
Majeur Entités Computer Vision enregistrées sans exposition API
Le domaine contient trois entités Computer Vision (DetectionResult, UIElementCache, UIPattern) avec leurs DbSet enregistrés dans le contexte. Aucun controller ne les expose. Ce code mort alourdit le schéma de base de données sans valeur fonctionnelle actuelle.
Solution recommandée : Soit implémenter les controllers correspondants, soit supprimer les entités et créer une migration de nettoyage.
Mineur Persistance/Services/Class1.cs — Fichier vide non supprimé
Un fichier Class1.cs vide généré automatiquement par Visual Studio lors de la création du projet n'a jamais été supprimé. C'est un artefact sans impact fonctionnel, mais qui pollue l'arborescence et signale un manque de rigueur dans le nettoyage du code.
Solution recommandée : Supprimer le fichier et faire un commit de nettoyage.
Mineur Couverture de tests insuffisante
Les tests unitaires couvrent uniquement le domaine TestSuites (Commands + Queries + Validators). Les features Auth, Projects, AI Agent, NLP Mappings et le DashboardRepository ne sont pas testés. Pour un produit de production, une couverture minimale de 70–80% serait attendue.
Solution recommandée : Étendre les tests aux handlers Auth et Projects. Ajouter des tests d'intégration avec TestContainers (PostgreSQL réel dans Docker au moment des tests).
Mineur Pas de versioning de l'API
L'API ne dispose pas de mécanisme de versioning ([ApiVersion], header api-version, ou segment d'URL /v1/). Toute modification breaking change casse immédiatement les clients existants sans garde-fou de compatibilité.
Solution recommandée : Intégrer le package Asp.Versioning.Mvc pour le versioning par URL ou par header. Permettre aux clients de cibler une version stable.

Architecture & Conception
1
Pourquoi avez-vous choisi la Clean Architecture plutôt qu'une architecture MVC classique ?
Réponse complète

La Clean Architecture satisfait plusieurs contraintes clés du projet :

  • Testabilité : La couche Application ne dépend d'aucun framework (ni EF Core, ni ASP.NET), permettant de mocker facilement les repositories dans les tests unitaires.
  • Maintenabilité : Chaque couche a une responsabilité unique. Changer de base de données (PostgreSQL → SQL Server) ne touche que la couche Persistance, sans impacter l'Application ni le Domain.
  • Évolutivité : L'ajout d'une nouvelle fonctionnalité suit un chemin clair — Command/Query dans Application, Controller dans API — sans modifier le noyau métier.

MVC classique aurait couplé la logique métier aux controllers, rendant les tests et la maintenance difficiles à mesure que la plateforme grandit.

2
Expliquez le pattern CQRS et comment vous l'avez implémenté.
Réponse complète

CQRS (Command Query Responsibility Segregation) sépare les opérations en deux catégories :

  • Commands : Modifient l'état de la base de données. Ex: AddProjectCommand, DeleteScenarioCommand.
  • Queries : Lisent l'état sans le modifier. Optimisables avec .AsNoTracking(). Ex: GetAllProjectsQuery.

Implémentation via MediatR : Chaque Command/Query est un record C# implémentant IRequest<ResponseHttp>. Le Handler est une inner class du même fichier. Le controller envoie simplement await _mediator.Send(cmd) sans connaître l'implémentation.

Avantage concret : La logique métier est testable indépendamment des controllers HTTP. On peut optimiser les queries (read-only, AsNoTracking) sans risquer d'impacter les commands.

Sécurité & Authentification
3
Comment fonctionne votre système d'authentification ? Est-il sécurisé ?
Réponse complète

Flux : L'utilisateur envoie email + mot de passe → BCrypt.Verify() valide le hash → JWT HS256 généré avec email et userId dans les claims → retourné au client.

Points forts :

  • BCrypt avec work factor 10 — résistant aux attaques par dictionnaire
  • JWT signé HS256 — intégrité garantie, infalsifiable sans la clé secrète
  • Rate limiting sur le login (5 requêtes/minute) — protection anti-brute-force
  • OAuth Google/GitHub — on ne stocke jamais les mots de passe pour ces utilisateurs
  • Vérification email à l'inscription — empêche les comptes fictifs

Limitation transparente : Le refresh token n'est pas implémenté (HTTP 501). Les tokens expirent et l'utilisateur doit se reconnecter. Pour un produit de production, un mécanisme de rotation de refresh tokens serait indispensable.

4
Comment avez-vous géré la gestion des rôles et des permissions ?
Réponse complète

La gestion des rôles est à deux niveaux :

  • Rôle global (Admin, Manager, Tester, Viewer) : associé à l'utilisateur via la table UserRoles.
  • Appartenance projet : via la table ProjectMembers — un utilisateur est membre ou non d'un projet, indépendamment de son rôle global.

Logique de permission :

  • Admin : voit et gère tout, sans restriction de projet
  • Manager : gère uniquement les projets dont il est membre, peut créer de nouveaux projets
  • Tester/Viewer : accès lecture seule aux projets dont ils sont membres

La vérification est faite côté serveur dans chaque endpoint, via des méthodes privées dédiées (CanReadProjectAsync, CanWriteProjectAsync) qui interrogent la base de données.

5
Pourquoi le rôle n'est-il pas dans le JWT ? N'est-ce pas une erreur de conception ?
Réponse complète

C'est une décision de conception discutable avec des avantages et inconvénients :

  • Avantage principal : Si le rôle d'un utilisateur change, la permission est effective immédiatement à la prochaine requête — pas besoin d'invalider le token existant.
  • Inconvénient majeur : Une requête SQL supplémentaire par endpoint protégé pour récupérer le rôle. Pour une plateforme à fort trafic, c'est un coût non négligeable.

Alternative recommandée pour une v2 : Inclure le rôle dans le JWT + un timestamp role_assigned_at. À la vérification, comparer avec la date de dernier changement de rôle pour invalider si nécessaire. Alternativement, utiliser Redis comme cache de session avec un TTL court (5 min).

Intégrations & Déploiement
6
Comment fonctionne l'intégration avec l'agent IA ? Que se passe-t-il si l'agent est indisponible ?
Réponse complète

L'intégration est via un service HTTP typé (IAAgentService) qui appelle le microservice FastAPI Python :

  1. Le frontend envoie une requête POST /api/ai-runs avec l'ID du scénario
  2. IAAgentService construit une requête HTTP vers http://localhost:8000/run avec un timeout de 5 minutes
  3. L'agent Python exécute le test (YOLOv8 pour la détection UI, NLP pour les mappings, automation browser)
  4. Le résultat JSON est désérialisé et persisté en base (TestExecution, TestResult, StepResult)

Si l'agent est indisponible : HttpClient lève une HttpRequestException ou un timeout après 5 minutes. La limitation actuelle est l'absence de circuit breaker — une amélioration serait d'intégrer Polly avec un circuit breaker pour échouer rapidement (fail-fast) et retourner un message explicite au lieu d'attendre le timeout complet.

Comment Entity Framework Core gère-t-il les migrations ? Avez-vous eu des problèmes ?
Réponse complète

EF Core génère les migrations automatiquement depuis les entités du domaine via dotnet ef migrations add NomMigration. Le projet compte 7 migrations correspondant à l'évolution itérative du schéma :

  • Migration initiale (schéma complet de toutes les entités)
  • Ajout des champs de reset de mot de passe (ResetToken, ResetTokenExpiry)
  • Ajout du champ Color pour les tags
  • Ajout du champ Priority sur les mappings NLP
  • Ajout des champs de vérification email (EmailVerificationToken)
  • Renommage Owner → Admin (rôle)
  • Ajustements de schéma divers

Défis rencontrés : Les migrations sur des tables avec des relations complexes peuvent nécessiter d'ajuster manuellement l'ordre des opérations pour éviter des violations de contraintes de clé étrangère. La commande dotnet ef database update est utilisée pour appliquer les migrations.

Patterns & Outils techniques
8
Qu'est-ce que FluentValidation et comment l'avez-vous intégré ?
Réponse complète

FluentValidation est une bibliothèque de validation déclarative pour .NET. Au lieu d'éparpiller des validations if (string.IsNullOrEmpty(...)) partout, les règles sont centralisées dans une classe Validator dédiée :

public class AddUserLoginValidator : AbstractValidator<AddUserLogin>
{
    public AddUserLoginValidator()
    {
        RuleFor(x => x.Email)
            .NotEmpty().EmailAddress();
        RuleFor(x => x.Password)
            .NotEmpty().MinimumLength(6);
    }
}

Double intégration dans le projet :

  • Dans les controllers Auth : validation manuelle avec validator.ValidateAsync() avant _mediator.Send()
  • Via le pipeline MediatR avec ValidationBehavior<TRequest, TResponse> : automatique pour tous les autres commands/queries

Les validators sont enregistrés automatiquement par scan d'assembly via services.AddValidatorsFromAssembly().

Comment fonctionne votre système d'audit logging ?
Réponse complète

Le AuditLogMiddleware est un middleware post-response qui s'exécute après chaque requête HTTP. Il enregistre automatiquement toutes les actions de modification :

  • Déclenché si : requête authentifiée + méthode non-GET + code réponse < 500
  • Ignoré si : l'endpoint est /api/dashboard/audit-logs (anti-récursion infinie)
  • Données enregistrées : userId, action (POST/PUT/DELETE), type d'entité extrait de l'URL, timestamp UTC, adresse IP

Pattern DI correct : Le middleware crée un nouveau scope DI (serviceProvider.CreateScope()) pour résoudre le DbContext — pattern indispensable car les middlewares sont des singletons alors que le DbContext est un service scopé.

Les logs sont accessibles via GET /api/dashboard/audit-logs avec pagination (Admin uniquement).

Qu'est-ce que le pattern Repository et pourquoi l'avez-vous utilisé ?
Réponse complète

Le pattern Repository abstrait l'accès aux données derrière une interface. Au lieu qu'EF Core soit utilisé directement dans les handlers, on passe par IProjectRepository dont l'implémentation concrète est dans la couche Persistance.

Avantages dans ce projet :

  • Testabilité : On peut mocker IProjectRepository dans les tests unitaires sans toucher la base de données réelle
  • Découplage : La couche Application ignore qu'on utilise EF Core — on pourrait passer à Dapper sans modifier l'Application
  • CRUD générique : GenericRepository<T> fournit Add, Update, Delete, GetById, GetAll pour tous les types d'entités
  • Repositories spécialisés : DashboardRepository n'hérite pas de GenericRepository car il agrège des données de plusieurs tables avec des requêtes complexes

Quelle est la différence entre une commande et une requête dans votre CQRS ?
Réponse complète

Le principe fondamental du CQRS est que lire et écrire sont des opérations fondamentalement différentes qui méritent des chemins de code séparés :

  • Command : A un effet de bord — modifie l'état persisté. Retourne généralement un identifiant ou un statut de confirmation. Ex: AddProjectCommand, DeleteScenarioCommand
  • Query : Sans effet de bord — lit l'état sans le modifier. Peut être optimisée avec .AsNoTracking() et des projections directes. Ex: GetAllProjectsQuery, GetDashboardSummaryQuery

En pratique dans le code : les deux sont des record : IRequest<ResponseHttp>. MediatR les achemine vers leur handler respectif via le type du record. La convention de nommage (Command vs Query en suffixe) suffit à les distinguer explicitement.

Avez-vous pensé à la scalabilité ? Comment scaler votre application ?
Réponse complète

La Clean Architecture facilite plusieurs axes de scalabilité :

  • Scalabilité horizontale : Le backend ASP.NET Core est stateless (JWT, pas de session serveur) — on peut lancer plusieurs instances derrière un load balancer sans configuration spéciale.
  • Mise en cache : IMemoryCache est enregistré en DI. Pour un cache distribué multi-instances, basculer vers Redis requiert uniquement de changer l'implémentation de l'interface, sans toucher la couche Application.
  • Agent IA indépendant : Le microservice Python peut être scalé indépendamment (Kubernetes HPA basé sur la charge des exécutions de tests).

13
Comment l'intégration OAuth fonctionne-t-elle sans exposer le secret côté client ?
Réponse complète

Le flux OAuth mis en œuvre est un Authorization Code Flow avec échange serveur à serveur :

  1. Angular redirige l'utilisateur vers Google/GitHub avec le client_id et le redirect_uri
  2. Après approbation, Google/GitHub retourne un code de courte durée au frontend
  3. Angular envoie ce code au backend (POST /api/auth/google/callback)
  4. Le backend échange le code contre le profil utilisateur en utilisant son client_secret
  5. Le backend crée ou retrouve l'utilisateur, génère un JWT interne et le retourne au client

Principe de sécurité : Le client_secret OAuth n'est jamais envoyé au navigateur — uniquement le backend l'utilise pour l'échange de code. Le navigateur ne voit jamais un secret OAuth.

  • Performances analytiques : PostgreSQL excelle sur les requêtes analytiques complexes (dashboard, tendances) grâce à son planificateur de requêtes avancé
  • Docker-first : Image officielle légère et de confiance, idéale pour un déploiement containerisé (docker-compose.yml inclus)
  • JSONB natif : Support natif du JSONB pour stocker des données semi-structurées (résultats IA) si nécessaire, sans schema migrations complexes

📄 Pattern interne d'un Feature — Convention inner-class Handler

Dans ce projet, chaque fichier Command contient à la fois le record Command ET son Handler (inner class). Ce pattern est une convention qui évite de jongler entre plusieurs fichiers.

C# — Structure type d'un Command (ex: AddTestSuiteCommand.cs) 
namespace Application.Features.TestSuitesFeature.Commands;

// 1. Le record Command (les données en entrée)
public record AddTestSuiteCommand(
    Guid ProjectId,
    string Name,
    string? Description,
    Guid UserId
) : IRequest<Guid>;  // retourne le Guid du nouvel objet

// 2. Le Validator FluentValidation
public class AddTestSuiteCommandValidator : AbstractValidator<AddTestSuiteCommand>
{
    public AddTestSuiteCommandValidator()
    {
        RuleFor(x => x.Name).NotEmpty().MaximumLength(200);
        RuleFor(x => x.ProjectId).NotEmpty();
    }
}

// 3. Le Handler (inner class — contient la logique)
public class Handler : IRequestHandler<AddTestSuiteCommand, Guid>
{
    private readonly ITestSuiteRepository _repository; // interface, pas EF Core
    private readonly ITestTestAutoumatisationContext _context;

    public Handler(ITestSuiteRepository repo, ITestTestAutoumatisationContext ctx)
        => (_repository, _context) = (repo, ctx);

    public async Task<Guid> Handle(AddTestSuiteCommand cmd, CancellationToken ct)
    {
        // Vérification métier (projet existe ?)
        var proj = await _context.Projects.FindAsync(cmd.ProjectId, ct);
        if (proj == null) throw new InvalidOperationException("Project not found");

        // Construction de l'entité Domain (POCO pur)
        var suite = new TestSuite { Name = cmd.Name.Trim(), ProjectId = cmd.ProjectId };

        // Persistence via interface (découplé de EF Core)
        await _repository.AddAsync(suite);
        return suite.Id;
    }
}

📁 Application/Behaviors/ — Pipeline MediatR

Pour réécrire le projet from scratch — par où commencer ?

Ordre recommandé :
1. Domain : Créer les entités (POCOs) et les interfaces — aucune dépendance externe.
2. Application : Créer les Commands/Queries/Validators/Handlers — référencer uniquement les interfaces Domain.

4. API : Câbler tout en DI dans Program.cs, créer les Controllers qui délèguent à MediatR.
5. UnitTest : Mocker les interfaces Domain et tester les Handlers en isolation.
